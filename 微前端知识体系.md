
## 领域概述

- 微前端：把“一个前端应用”拆成多个可独立开发、独立发布、运行时组合的子应用（或模块），由基座负责路由分发、资源加载、公共能力与治理。
- 典型收益：团队自治、渐进式迁移、按业务域拆分、独立发布；典型成本：运行时复杂度、性能与稳定性、治理与规范、跨应用协作。
核心知识体系（结构化大纲）

1. 基本概念与边界（What/Why）
- 1.1 微前端定义与非目标（★★★｜★★★｜基础｜理论）：不是“多页应用”、不是“组件库拼装”；关注“独立交付+运行时集成”
- 1.2 单体/多 SPA/微前端对比（★★★｜★★★｜基础｜理论）：组织结构、部署粒度、依赖管理、故障域
- 1.3 适用/不适用场景（★★★｜★★★｜进阶｜项目）：强自治团队、多业务域、遗留系统迁移；小团队/强耦合交互不划算
- 1.4 子应用“边界”设计：业务域、路由域、数据域（★★★｜★★｜进阶｜项目）
- 1.5 成功指标：交付效率、故障隔离、性能预算、治理成熟度（★★｜★★｜进阶｜项目）
2. 架构形态与集成方式（How to compose）
- 2.1 构建时集成 vs 运行时集成（★★★｜★★★｜基础｜理论）：Module Federation 属于运行时模块共享但偏构建体系配合
- 2.2 路由分发型（基座按 URL 激活）（★★★｜★★★｜进阶｜项目）：最常见、治理点最多
- 2.3 组件/模块级微前端（★★｜★★｜高级｜项目）：共享依赖、版本冲突更敏感
- 2.4 iFrame 方案（★★｜★★★｜基础｜项目）：隔离强、体验与通信受限、SEO/性能权衡
- 2.5 Web Components/Custom Elements（★★｜★★｜进阶｜编码）：框架无关、工程与生态成本
- 2.6 Server-side composition（★｜★｜高级｜项目）：边缘拼装/服务端模板拼装的前后端协同
3. 运行时机制（加载、挂载、生命周期）
- 3.1 资源加载与沙箱执行（★★★｜★★★｜进阶｜编码）：HTML Entry、脚本注入、执行上下文
- 3.2 生命周期：bootstrap/mount/unmount/update（★★★｜★★★｜进阶｜编码）：状态清理、重复挂载、幂等
- 3.3 多实例/预加载/缓存策略（★★｜★★｜高级｜优化）：预取、按需、保活 keep-alive 的取舍
- 3.4 失败策略：加载失败降级、重试、熔断（★★★｜★★｜高级｜项目）
- 3.5 可观测性：埋点、日志、分布式追踪、版本标识（★★｜★★｜高级｜项目）
4. 隔离与共享（最常考的“工程真问题”）
- 4.1 JS 隔离（★★★｜★★★｜高级｜项目）：全局污染、原型链修改、事件监听泄露、定时器泄露
- 4.2 CSS 隔离（★★★｜★★★｜进阶｜项目）：BEM/命名空间、CSS Modules、Shadow DOM、运行时样式隔离
- 4.3 依赖共享与冲突（★★★｜★★★｜高级｜项目）：React/Vue 多版本、单例依赖、peerDependencies、externals
- 4.4 静态资源与公共路径（★★★｜★★★｜进阶｜项目）：base/publicPath、CDN、多环境配置
- 4.5 权限与公共能力共享（★★★｜★★｜进阶｜项目）：鉴权、用户态、特性开关、灰度/AB
5. 路由与导航协同（用户体验核心）

- 5.3 跨应用跳转与状态保留（★★★｜★★｜高级｜项目）：面包屑、tabs、多窗口

- 8.2 运行时性能：沙箱/样式隔离/事件代理开销（★★★｜★★｜高级｜优化）

知识点关联（网状理解，而非线性背诵）

- 集成方式(2) 决定 运行时机制(3) 的复杂度；运行时机制(3) 直接带来 隔离与共享(4) 的问题
- 路由协同(5) 与 通信状态(6) 往往绑定：跨应用跳转需要传参与状态恢复
- 工程化治理(7) 是把(4)(6)(8)“制度化”的手段：没有治理，微前端会退化为“更难维护的大单体”
- 性能稳定安全(8) 是对(2)(3)(4)(7)的验收标准：能不能扛住线上复杂度
高频面试考点（★★★项 + 常见问法 + 回答要点）

- 微前端的核心价值与代价是什么？（★★★）
  - 要点：独立交付/团队自治/渐进迁移；代价是运行时复杂度、治理成本、性能与稳定性风险
- 你会如何做 CSS/JS 隔离？各自的坑是什么？（★★★）
  - 要点：CSS 命名空间/CSS Modules/Shadow DOM；JS 沙箱/全局污染治理；卸载清理与副作用回收
- 基座与子应用路由怎么设计？刷新与 404 怎么处理？（★★★）
  - 要点：路由主导权、前缀/ basename、history 回源、服务端 rewrite、子应用激活规则
- 子应用之间怎么通信？如何避免“全局事件总线失控”？（★★★）
  - 要点：通信分层（配置/事件/状态/URL）；事件命名空间、类型约束、订阅释放、版本兼容
- 依赖如何共享与版本治理？为什么会出现 React 多实例问题？（★★★）
  - 要点：单例依赖、externals/共享模块、peerDependencies、版本锁定与回滚策略
- 一个子应用加载失败或运行时报错，怎么兜底？（★★★）
  - 要点：超时/重试/熔断、降级页面、错误边界、监控告警、灰度回滚
面试深度分层（考察重点）

- 一面：概念边界、场景判断、路由与通信基本方案、隔离基本手段
- 二面：版本治理与依赖冲突、工程化落地（注册/发布/灰度/回滚）、性能预算与预加载策略
- 三面：组织与治理体系、跨团队规范、容灾与安全、复杂交互与长期演进（从单体迁移到微前端）
学习路径建议（从能用到能治理）

- 阶段 1（入门可落地）：理解形态差异(1)(2) → 路由协同(5) → 基础通信(6.1/6.2)
- 阶段 2（项目可上线）：隔离与共享(4) → 失败兜底与可观测(3.4/3.5) → 本地联调与发布(7.5/7.2)
- 阶段 3（规模化治理）：注册发现(7.1) + 质量门禁(7.6) + 性能稳定安全(8) + 规范体系(7.4)
